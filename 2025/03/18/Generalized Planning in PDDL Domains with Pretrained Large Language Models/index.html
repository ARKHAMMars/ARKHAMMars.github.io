前置知识：

PDDL:

一种用于描述规划问题领域的形式化语言。它是人工智能领域中用于描述规划问题的一种常见语言。PDDL主要用于描述有关规划问题的领域知识，包括问题的初始状态、目标状态、操作符、约束条件等。PDDL通常被用来描述问题的初始状态和目标状态，然后通过规定一系列可用的操作符以及这些操作符对状态的影响来解决问题。\
PDDL通常由以下几个部分组成：

  1. 声明（Declarations）：包括定义对象、谓词和函数等。
  2. 初始状态（Initial State）：描述问题的初始状态，即问题开始时各个对象的状态。
  3. 目标（Goals）：描述问题的目标状态，即问题需要达到的状态。
  4. 操作符（Actions）：定义可用的操作符或动作，以及这些操作符对状态的影响。
  5. 约束条件（Constraints）：描述解决问题时需要满足的条件。\
    使用PDDL可以帮助研究人员更清晰、准确地描述规划问题的领域知识，有助于计算机系统理解和解决这些问题。

Generalized Planning程序特征

  1. 带有条件和循环的规划/程序
    这是一种较常见的表示方式：最终的广义规划可写成类似伪代码或小型程序的形式，其中包含：

    • if-then-else 条件分支
    • while 或 for 循环
    • 对特定情形的处理策略
      这种形式易于理解并具有较强的表达能力，能够嵌入复杂逻辑，适用于在不同状态下执行不同操作。

  2. 有限状态机（Finite State Machine, FSM）或控制器
    有些研究会将广义策略表示为一个有限状态机。其中每个状态对应某种“决策子策略”或“控制模式”，依据当前环境观测和状态机中的转移条件（如“是否已经搬运完所有箱子”）进行状态跳转，从而完成不同实例下的一系列动作。

  3. 层次化任务网络（HTN）
    在分层规划（Hierarchical Planning）中，可以用HTN（Hierarchical Task Network）来表示更高层次的任务分解与子任务结构，如果能找到对多个实例都适用的一套分解方案（即一套“广义”HTN规划模板），也可以看作是一种广义规划的表达。

  4. 强化学习和策略学习的结合
    在一些更贴近机器学习（尤其是强化学习）的场景下，会把Generalized Planning的思路整合进策略学习当中，希望训练出一个能够处理不同规模、不同初始条件的“政策网络（Policy Network）”或“策略函数”。

    • 这类方法往往需要大量训练数据或模拟环境，用学习到的策略覆盖尽可能多的任务实例。

论文梳理

目的

探究大语言模型是否能作为广义规划器：给定一个领域和训练任务，生成一个程序，该程序能高效地为该领域内的其他任务生成规划。

方法

使用规划域定义语言（PDDL）描述的领域，并利用GPT-4来合成Python程序。

  • 思维链（CoT）总结，即促使大语言模型（LLM）在合成程序之前，先用语言总结该领域并提出策略
  • 自动调试，即根据训练任务对程序进行验证，若程序出现错误，会依据四种反馈类型重新向大语言模型发出提示。

PDDL域：

PDDL域中的广义规划

在训练阶段，会从这些任务分布中选取一小部分训练任务（在实验中通常为 10 个或更少）。而一组预留的评估任务（通常涉及更多的对象）则用于衡量性能。

目标：

  • 利用训练任务来合成一个程序，该程序要能为所有评估任务生成有效的规划方案。
  • 程序能在固定的时间预算内（实验中为 30 秒）为评估任务返回一个有效的规划，认为这个评估任务得到了解决。
  • 关注的是能够满足要求的规划，而非最优规划

自动交互式调试

利用训练任务来验证这一实现方案：

执行获取规划的操作，直到它返回一个输出结果、抛出异常或者达到超时时间（30 秒）。

  • 输出的是一个有效的规划，我们就继续处理下一个训练任务。
  • 否则，我们会根据以下四种反馈类型中的一种来重新给出提示

“Python Exceptions.”

如果执行 “获取规划（get plan）” 操作导致了 Python 异常，我们会捕获追溯信息，并将其与输入内容一起反馈给大语言模型（LLM）

“Timeout.”

如果 “获取规划（get plan）” 操作在超时时间之前未能完成，我们会告知大语言模型（LLM）该程序未完成，并提示可能存在无限循环的问题。

“Plan Syntax.”

如果 “获取规划（get plan）” 操作返回了一个输出结果，我们会检查其语法：检查它是否是一个字符串列表，每个字符串是否都包含在括号内且以空格分隔，以及每个操作的操作名称、对象名称和每个操作对应的对象数量，相对于该领域和任务而言是否有效。

“Plan Semantics.”

用 VAL 工具（豪伊、朗和福克斯，2004 年）来检查 “获取规划” 的输出是否是一个语义上有效的规划。\

“Additional Details.”

  • 重复检查代码并报告任何错误的过程，最多重复四次。
  • 将大语言模型的每一个新回复添加到一个不断增长的 Python 文件中，而不是覆盖之前的回复。
  • 最后一次尝试时仍然遇到错误，那么在评估时将使用最终的回复

论文精读

科学问题

相关方法及其局限性：

  1. ​传统广义规划的局限性

    • ​缺陷1：策略表示能力受限
      经典广义规划方法（如PG3）依赖人工设计的符号化策略模板（如决策列表、有限状态机），难以表达复杂领域逻辑（如递归操作、动态对象集合遍历）。

    • ​缺陷2：搜索效率低下
      策略空间的组合爆炸导致传统方法无法处理大规模任务（如超过100个对象的物流配送问题），时间复杂度随对象数量呈指数增长。

  2. ​LLM直接用于规划的不足

    • ​缺陷3：生成结果不可控
      LLM直接生成计划序列时，可能违反PDDL语义约束（如忽略操作前提条件），导致无效计划。

    • ​缺陷4：逐任务规划的重复性
      LLM为每个任务独立生成计划，无法复用领域知识，导致计算冗余（如相同子问题重复求解）。

  3. ​符号化规划与学习的割裂

    • ​缺陷5：符号方法的僵化性
      纯符号化规划（如PDDL）依赖人工编码领域知识，难以自适应新场景（如动态增减配送节点）。

    • ​缺陷6：学习方法的黑盒性
      基于强化学习（RL）的方法缺乏可解释性，难以验证其策略是否符合领域逻辑。

  4. ​验证方法的不足

    • ​缺陷7：人工验证不可靠
      传统广义规划依赖人工测试有限任务实例，无法保证生成策略的通用性。

鉴于上述方法的缺陷，故而结合LLM和广义规划来生成领域内的广义规划。

核心问题：

  1. 如何引导LLM正确理解特定领域内的设定，不发生差错
  2. 如何确保LLM生成的代码符合问题逻辑
  3. 如何引导LLM进行自主的debugging环节
  4. 面对复杂问题和LLM窗口大小有限的冲突问题如何进行处理
  5. 验证各个组成部分对于结果的影响，寻找重要的组成部分(消融实验即可）

解决方案

使用</span>PDDL域（Planning Domain Definition Language）​作为输入优势在于</span>形式化表达能力、</span>与经典规划系统的兼容性，以及</span>促进大型语言模型（LLM）的领域知识迁移（q1，q4)

形式化表达与结构化知识传递：

通过谓词逻辑（Predicates）​和动作模式（Actions）​明确定义领域内的状态、操作前提与效果。这种形式化特性使得LLM能够精确理解任务逻辑，而非依赖模糊的自然语言描述。一定程度上解决了问题1.

与经典规划系统的兼容性

经典符号规划器的标准输入格式，将其作为LLM的输入可实现混合架构协同：

  • 有利于执行验证与调试支持，可以从语法上进行检查
  • 使用PDDL能够直接和PG3的对比，验证优势

提升LLM的领域泛化能力

可聚焦于</span>策略抽象而非底层搜索细节，从而突破传统规划的局限性</span>

  • 抑制搜索倾向

  • 对象无关性适配：PDDL的变量参数化（如</span>?block）使LLM生成的程序能处理任意规模任务（如3块或1000块积木的堆叠问题）</span>

应对窗口大小问题

基于PDDL以两种方式缩写训练任务的编码

  • 首先，我们总是只使用两个训练任务，即使给出了更多任务。
  • 其次，在每个训练任务中，我们限制显示的对象数量和初始状态基础原子的数量。

标准化与可扩展性优势

PDDL的应用为连接LLM与规划领域的</span>理想桥梁：</span>

  • 领域扩展便捷：新增领域仅需提供PDDL描述，无需重构模型架构（如自动驾驶、机器人导航等场景）</span>

  • 研究可复现性：PDDL的公开基准库（如IPC竞赛领域）便于其他研究者复现实验，验证方法的普适性</span>

对比喻其他的基于指针的编程语言表示等输入方式，整体更加有利于自动化的形成我们所需要的标准化、兼容符号规划器

使用CoT的思想，形成了合理的混合架构的框架(q2)

领域理解的显式结构化

通过引入PDDL（Planning Domain Definition Language）的形式化输入框架，研究构建了面向大语言模型的</span>结构化认知引导机制。在输入阶段强制要求模型对PDDL领域进行自然语言总结，这一设计实现了：</span>

  1. 语义锚定：将符号化逻辑谓词（如(isHomeBase ?l)）映射为可解释的领域语义（如”核心枢纽位置”），避免模型陷入语法解析的歧义陷阱；

  2. ​增加语义抽象层稳定性：通过分步的”总结-策略”流程，将混合的符号系统解构为层次化知识表示，如图1所示：

    • ​PDDL符号层：原始领域定义与任务实例

    • ​语义抽象层：自然语言提炼的核心逻辑

    该步骤显著提升了模型对形式化规划问题的系统性理解能力，完成对。实验数据显示（见表1），在对象类型超过10种、谓词关系复杂的Heavy域，CoT引导使模型准确识别出隐含的拓扑排序需求，成功率较基线提升60%。这种结构化认知框架有效规避了LLM自主学习的随机性风险，为符号系统与神经模型的深度融合提供了新范式。

策略描述

将代码生成流程重构为PDDL优先的双阶段架构，显著提升生成过程的可解释性与可靠性：

  1. ​PDDL中间层生成

    • ​结构化逻辑表达：强制LLM首先输出PDDL领域文件（domain.pddl）与问题解决方案（problem.pddl），通过形式化语言约束策略表达边界

    • ​语义一致性检查：利用PDDL语法检查工具（如VAL）进行前置验证，捕捉谓词定义错误、动作前提矛盾等基础错误

    • 搜索抑制机制：在prompt中明确禁止使用BFS/DFS等搜索算法，强制LLM使用别的算法。

  2. ​代码转换阶段

    • ​模式化翻译规则：基于验证通过的PDDL文件，生成符合”规划即程序”范式的Python代码

  • ​语义解析深度：在Heavy域中，GPT-4成功识别(heavier ?x ?y)谓词的传递闭包特性，生成基于拓扑排序的堆叠策略（代码清单7），而GPT-3.5完全失败（表1成功率0%）

基于CoT框架的四类错误精准定义与分层诊断(q3)

  2. 超时错误（算法复杂度失控）
  3. 计划语法错误（PDDL-动作映射失准）
  4. 计划语义错误（逻辑一致性崩坏）​

调试效率的指数级提升，帮助自我交互调试实现

通过错误类型与修复策略的精确映射，建立自动化调试的决策树</span>

  • 在Delivery域中，将平均调试迭代次数从6.2次​（未分类时）降至2.1次

模型弱点的可解释性诊断

错误类型分布揭示LLM在不同认知层级的缺陷，四种错误定义覆盖了这一点。帮助LLM更加快速决定在哪里重新思考。</span>

能力维度映射：

  3. 实验流水线：实验过程全自动化，唯一需要手动操作的部分是将提示和响应从终端复制并粘贴到浏览器中。

复现难度

数据集复现（中等）

  • 公开数据集：前六个PDDL领域的数据集来自于论文PG3: Policy-Guided Planning for Generalized Policy Generation，并且在GitHub上公开（GitHub链接），可以方便地获取，并进行筛选进行实验。
  • heavy数据集：这是一个由作者自行创建的未公开数据集，复现时需要联系原作者获取，或根据任务设置构造类似数据集，增加复现难度。
  • 任务设定：每个领域包含10个训练任务和30个评估任务，复现时可以根据论文描述进行明确的任务设置，较易实现。

代码复现（简单）

  • 公开代码：实验代码已在GitHub公开，下载并按照说明进行配置微调即可复现。
  • 依赖配置：会遇到一些依赖或环境配置问题，在不同硬件和操作系统上运行时，需要调试和适配。

实验环境复现（简单）

  • 硬件配置：实验使用M1芯片和64GB内存的Macbook Pro。算力要求并非很高。
  • 软件接口：由于GPT-4的API不可用，实验通过ChatGPT浏览器接口进行。
  • 手动操作：尽管实验流水线是自动化的，但仍需要手动操作复制和粘贴提示与响应，会影响复现的整体效率。

总而言之，整体复现难度中等

LLM与经典规划系统的桥梁

使用PDDL域作为输入，使LLM能理解广义规划问题
在本研究中，我们采用PDDL（Planning Domain Definition Language）域作为输入，这是一种标准化的语言，用于描述规划问题的领域和任务。通过将PDDL域作为输入，LLM能够解析和理解广义规划问题中的结构和约束。PDDL的形式化定义为LLM提供了一个清晰的框架，使其能够处理规划任务中的各种因素，并为后续的推理和问题求解提供基础。

用广义规划问题作为输入，解决LLM的窗口大小问题
传统的LLM在处理较长文本时可能面临窗口大小的限制，即其输入序列的长度受限。通过将广义规划问题作为输入，LLM能够处理更复杂、具有多层次结构的任务。广义规划问题的设计本身就具有一定的层次性和模块化，使得模型可以有效分解任务并在较小的输入窗口内进行推理，从而克服了窗口大小对模型推理能力的制约。

用CoT思想引导LLM进行合理的推理，符合广义规划
为了确保LLM在解决广义规划问题时能够遵循合理的推理路径，我们引入了链式推理（Chain-of-Thought, CoT）思想。CoT通过逐步展开推理过程，帮助LLM在复杂任务中保持思维的连贯性和逻辑性。通过引导LLM生成一系列中间推理步骤，CoT使得模型能够清楚地展示每个推理阶段，确保最终输出的解答符合广义规划问题的要求和经典规划的推理规范。

以广义规划为最终目标，通过广义规划的解来组合求解复杂经典规划问题
最终，广义规划问题的解为我们提供了一个有力的工具来解决经典规划问题。通过将广义规划问题的解作为基础，LLM能够有效组合各个子任务的解，进而求解更加复杂的经典规划问题。这种方法利用广义规划提供的结构化信息，帮助经典规划问题的求解过程更加高效和灵活，展现了LLM在复杂推理任务中的优势。
